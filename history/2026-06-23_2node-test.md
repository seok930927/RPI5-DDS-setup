# 작업 히스토리 — 2노드 실통신 테스트 (시나리오 1-1)

- **날짜**: 2026-06-23
- **목적**: RPI4(Publisher) ↔ RPI5(Subscriber) 서로 다른 두 기기 간 Fast DDS 실통신 검증 (프로젝트 본 목표)
- **결과**: ✅ **성공** — RPI4 송신 20개 → RPI5 수신 20개 (유실 0)

---

## 구성

| 기기 | 역할 | IP | 실행 |
|------|------|-----|------|
| RPI4 | Publisher | 192.168.11.6 | `examples/helloworld/build/publisher` |
| RPI5 | Subscriber | 192.168.11.211 | `examples/helloworld/build/subscriber` |

- 같은 LAN, 도메인 0 → 멀티캐스트 Discovery로 자동 매칭 (별도 IP 설정 없음)

## 진행

1. **사전 준비 (RPI5)**: 2노드 테스트는 양쪽 실행 시차가 있으므로 subscriber 대기시간을 15초 → **5분(300초)**으로 늘려 재빌드 후 백그라운드 대기.
2. **네트워크 확인**: RPI5 → RPI4 ping 0% loss.
3. **실행**: RPI5 subscriber 먼저 대기 → RPI4 publisher 실행.

## 결과 (양쪽 대조)

| 항목 | RPI4 (Pub) | RPI5 (Sub) |
|------|-----------|-----------|
| 매칭 | 매칭 수 = 1 | 매칭 수 = 1 |
| 데이터 | 20개 송신 (index 1~20) | 20개 수신 |
| 유실 | — | 0 |
| 종료 | rc=0 | rc=0 |

- RPI4가 끝에 본 "매칭 수=0"은 RPI5가 20개 수신 후 먼저 종료한 정상 동작.
- RPI5 로그: `logs/09_2node_sub.log` (05:07:08 시작 → 05:07:53 종료)

## 결론

서로 다른 두 라즈베리파이 간 Fast DDS RTPS 통신이 멀티캐스트 Discovery → 엔드포인트 매칭 → 유니캐스트 데이터 전송까지 정상 동작함을 확인.
**시나리오 1-1 (PHASE 1 기본 통신) 통과.**
