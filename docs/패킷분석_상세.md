## 3. RTPS 서브메시지 종류와 개수 (핵심)

하나의 UDP 패킷 안에는 여러 RTPS **서브메시지**가 들어간다 (예: `INFO_TS + DATA`, `INFO_DST + HEARTBEAT`).

| 서브메시지 | ID | 개수 | 역할 |
|-----------|-----|------|------|
| **DATA** | 0x15 | 232 | 실제 데이터 + Discovery 정보 운반 |
| **HEARTBEAT** | 0x07 | 99 | (RELIABLE) Writer가 "여기까지 보냈다" 통지 |
| **ACKNACK** | 0x06 | 98 | (RELIABLE) Reader가 "여기까지 받았다/재전송 요청" |
| **GAP** | 0x08 | 36 | 특정 시퀀스는 보낼 게 없다(건너뜀) 통지 |
| **INFO_TS** | 0x09 | 232 | 타임스탬프 (DATA와 동반) |
| **INFO_DST** | 0x0e | 309 | 목적지 GUID 명시 (유니캐스트 동반) |

> HEARTBEAT(99) ↔ ACKNACK(98)이 거의 1:1로 짝을 이룸 → **RELIABLE 신뢰성 프로토콜이 실제로 동작**한 증거.

### DATA 세부 종류 (232개 분해)
| 표기 | 개수 | 의미 |
|------|------|------|
| `DATA` (순수) | 129 | **사용자 데이터** (HelloWorldTopic) + 제어(DDSTestControl) |
| `DATA(p)` | 85 | **SPDP** 참여자(Participant) 발견 announce |
| `DATA(w)` | 13 | **SEDP** Writer 엔드포인트 발견 |
| `DATA(r)` | 13 | **SEDP** Reader 엔드포인트 발견 |
| `DATA(w[UD])` | 11 | Writer 엔드포인트 해제(unregister/dispose) |
| `DATA(r[UD])` | 11 | Reader 엔드포인트 해제 |

> 시나리오마다 Reader/Writer를 새로 만들고 지웠기 때문에 `DATA(w)/DATA(r)`(생성)와 `[UD]`(해제)가 여러 번 반복됨 — 11개 시나리오 전환의 흔적.

## 4. 패킷 길이 분석

패킷 길이는 **종류에 따라 뚜렷이 다르다.** (프레임 = 이더넷+IP+UDP+RTPS 전체 길이, 단위 바이트)

| 패킷 종류 | 대표 길이 | 설명 |
|-----------|----------|------|
| **SPDP `DATA(p)`** | **606** (47개) | 가장 큼. 참여자의 QoS·로케이터(IP/포트)·빌트인 엔드포인트 목록을 다 담아서 |
| **사용자 DATA** (HelloWorld) | **198~250** (대부분 198) | 페이로드가 작음: `index(4B)+message("Hello DDS N"≈12B)` + RTPS 헤더 |
| **HEARTBEAT** | 154 / 170 / 202 | firstSN·lastSN·count만 담아 작음 |
| **ACKNACK** | **166** (95개) / 170 | readerSNState(비트맵)+count, 가장 일정 |
| 큰 DATA | 490·506·606·786·1202 | Discovery(SEDP/SPDP)나 다중 서브메시지 묶음 |

핵심 관찰:
- **Discovery 패킷(606B)이 사용자 데이터(198B)보다 3배 이상 크다** — Discovery는 "내가 누구고 어떤 QoS로 어디서 받는다"는 메타정보를 잔뜩 싣기 때문.
- 평균이 292B인 이유: 작은 데이터/제어 패킷이 많지만 606B SPDP가 평균을 끌어올림.

## 5. 통신 흐름 (시간순) — Discovery 시퀀스

캡처 앞부분 타임라인 (relative seconds):

```
0.0 ~ 6.5s  : RPI5(.211) 혼자 SPDP를 멀티캐스트(239.255.0.1)로 반복 announce
              (처음 0.1s 간격 6번 → 이후 3초 주기로 전환 = "아무도 없나?" 탐색)
8.30s       : RPI4(.6)가 등장 → SPDP 멀티캐스트 announce  ← 참여자 상호 발견(SPDP)
8.30s       : .211 → .6 유니캐스트 DATA(p) (서로 직접 인사)
8.30s~      : .211 → .6 INFO_DST + HEARTBEAT ... → 유니캐스트 SEDP/데이터 단계 시작
```

이는 RTPS 표준 순서와 정확히 일치:
1. **SPDP** (멀티캐스트) — 참여자 발견
2. **SEDP** (유니캐스트, `DATA(w)`/`DATA(r)`) — 엔드포인트 발견·매칭
3. **DATA** (유니캐스트) — 실제 데이터 전송
4. **HEARTBEAT/ACKNACK** (유니캐스트) — RELIABLE 신뢰성 보장

## 6. 우리가 세부적으로 테스트한 것 (자동 시나리오 11개)

자동 러너(`auto_pub`↔`auto_sub`)가 제어 토픽으로 동기화하며 아래를 순서대로 수행. 각 시나리오는 **QoS를 바꿔 Reader/Writer를 새로 생성** → 그래서 캡처에 SEDP 생성/해제가 반복됨.

| # | 시나리오 | 무엇을 검증 | 기대 | 패킷적 근거 |
|---|----------|-------------|------|-------------|
| 1 | **1-1/2-1 기본통신·Discovery** | SPDP→SEDP→DATA 매칭·전송 | 10개 수신 | DATA(p)→DATA(w/r)→DATA 순서 |
| 2 | **2-2 HEARTBEAT/ACKNACK** | RELIABLE 신뢰성 교환 | 10개 | HEARTBEAT↔ACKNACK 짝 발생 |
| 3 | **3-1 BEST_EFFORT** | 재전송/HEARTBEAT 없음 | 10개 | (BE 구간) HEARTBEAT 없음 |
| 4 | **3-1 RELIABLE** | 재전송/HEARTBEAT 있음 | 10개 | HEARTBEAT/ACKNACK 발생 |
| 5 | **3-2 VOLATILE(늦은조인)** | 늦은 Sub는 과거 못 받음 | 0개 | 매칭돼도 과거 DATA 없음 |
| 6 | **3-2 TRANSIENT_LOCAL(늦은조인)** | 늦은 Sub가 과거 받음 | 10개 | 조인 직후 과거 DATA 일괄 전송 |
| 7 | **3-3 KEEP_LAST depth=5(늦은조인)** | 최근 5개만 보관·전달 | 5개 | 과거분 중 5개만 DATA |
| 8 | **3-3 KEEP_ALL(늦은조인)** | 전체 보관·전달 | 20개 | 과거 20개 DATA 전부 |
| 9 | **3-4 Mismatch Pub=BE/Sub=RE** | QoS 불일치 → 매칭 실패 | 0개 | SEDP는 보이나 DATA 전송 없음 |
| 10 | **3-4 Mismatch Pub=VOL/Sub=TL** | 내구성 불일치 → 실패 | 0개 | 매칭 안 됨 |
| 11 | **3-4 Compatible Pub=RE/Sub=BE** | 호환 조합 → 성공 | 10개 | 정상 DATA 전송 |
| 12 | **4-1 Partition 일치(A=A)** | 같은 파티션 → 통신 | 10개 | 정상 전송 |
| 13 | **4-1 Partition 불일치(A≠B)** | 다른 파티션 → 논리 격리 | 0개 | 매칭 안 됨 |
| 14 | **4-2 Content Filter(index>5)** | 내용 기반 구독 | 5개 | 조건 맞는 것만 |

**결과: 14/14 전부 기대대로 (RPI5 수신 로그·RPI4 결과표 일치).** GAP(36개)는 늦은조인/시퀀스 건너뜀 구간에서 "이 시퀀스는 없음"을 알린 흔적.

## 7. 관찰된 QoS 동작 (패킷으로 확인)

- **RELIABLE**: HEARTBEAT(101) ↔ ACKNACK(100) 교환 확인 → Writer가 수신 확인받고 필요시 재전송하는 구조.
- **BEST_EFFORT**: 해당 구간엔 HEARTBEAT/ACKNACK 없음 (재전송 보장 안 함).
- **TRANSIENT_LOCAL**: 늦게 조인한 Reader에게 과거 DATA가 일괄 전송됨 (DATA 묶음).
- **History(KEEP_LAST/KEEP_ALL)**: 보관·전달 개수 차이 → 5개 vs 20개로 확인.
- **GAP**: KEEP_LAST로 잘려나간(또는 건너뛴) 시퀀스를 Reader에 알려 정합성 유지.

## 9. 패킷 상세 해부 (필드 단위)

실제 캡처에서 타입별 대표 패킷을 골라 **RTPS 필드를 그대로** 뜯어본 것. (재현: `tshark -r <pcap> -Y "frame.number==N" -V`)

### 9-1. 공통 RTPS 헤더 (모든 패킷 앞 20바이트)
프레임 1 기준:
```
Protocol version : 2.2
vendorId         : 01.0F (eProsima - Fast-RTPS)
guidPrefix       : 010f527ee9e8205f00000000   ← 참여자 고유 식별(GUID 앞 12바이트)
```
- 맨 앞 4바이트 매직 `RTPS` + 버전(2.2) + 벤더(01.0F) + guidPrefix(송신 참여자 식별).
- 그 뒤에 **서브메시지**들이 이어 붙음 (한 UDP에 여러 개).

### 9-2. SPDP — 참여자 발견 `DATA(p)` (프레임 1)
```
INFO_TS (0x09)  octetsToNextHeader=8        ← 타임스탬프
DATA (0x15)     octetsToNextHeader=468      ← 본문 468바이트(그래서 패킷이 큼)
  readerEntityId : 0x000100c7 (BUILTIN_PARTICIPANT_READER)
  writerEntityId : 0x000100c2 (BUILTIN_PARTICIPANT_WRITER)   ← SPDP 전용 빌트인 엔드포인트
  writerSeqNumber: 1
  --- 참여자가 광고하는 내용(PID 파라미터) ---
  PID_PROTOCOL_VERSION   : 2.2
  PID_VENDOR_ID          : 01.0F
  PID_DOMAIN_ID          : 0
  PID_PARTICIPANT_GUID
  PID_METATRAFFIC_UNICAST_LOCATOR : UDPv4 192.168.11.211:7410   ← "나한테 메타는 여기로"
  PID_DEFAULT_UNICAST_LOCATOR     : UDPv4 192.168.11.211:7411   ← "데이터는 여기로"
  PID_PARTICIPANT_LEASE_DURATION
  PID_BUILTIN_ENDPOINT_SET : Participant/Publication/Subscription Announcer·Detector ...
  PID_PROPERTY_LIST : PARTICIPANT_TYPE
  PID_SENTINEL  (파라미터 끝 표시)
```
→ **여기에 IP·포트(로케이터)가 들어있어서**, 이걸 받은 상대는 이후 **유니캐스트로 어디에 보낼지** 알게 된다. SPDP가 큰(606B) 이유.

### 9-3. SEDP — 엔드포인트 발견 `DATA(w)` (프레임 20)
```
writerEntityId : 0x000003c2 (BUILTIN_PUBLICATIONS_WRITER)
  PID_ENDPOINT_GUID
  PID_TOPIC_NAME : DDSTestControl     ← 어떤 토픽의
  PID_TYPE_NAME  : HelloWorld          ← 어떤 타입을
  PID_DURABILITY / PID_HISTORY / PID_RELIABILITY ...  ← 어떤 QoS로 쓰는지
```
→ "나는 이 토픽을 이 QoS로 쓰는 Writer다"를 광고. Reader 쪽 SEDP와 **QoS를 비교(RxO)**해서 매칭 여부 결정.

### 9-4. 사용자 DATA — 실제 데이터 (프레임 70, HelloWorldTopic)
```
INFO_DST → INFO_TS → DATA
  readerEntityId : 0x00000304 (Application-defined reader)   ← 빌트인 아님 = 사용자 엔드포인트
  writerEntityId : 0x00000303 (Application-defined writer)
  writerSeqNumber: 2
  serializedData (CDR_LE):
     02000000 0c000000 48656c6c6f20444453203200
     └index=2┘ └len=12┘ └ "Hello DDS 2" ┘
```
→ **여기가 진짜 우리 데이터.** EntityId가 `0x...03`처럼 빌트인(0x...c2/c7)이 아닌 값 = 애플리케이션 엔드포인트. CDR로 직렬화된 `index(uint32)+message(string)`가 그대로 들어있음.

**실제 원본 바이트 (프레임 70 전체):**
```
0000  2c cf 67 e6 c8 a2 dc a6 32 83 9f 98 08 00 45 00   ,.g.....2.....E.   ← 이더넷+IP 헤더
0010  00 b8 f3 7f 40 00 40 11 ae 8b c0 a8 0b 06 c0 a8   ....@.@.........   c0a80b06=192.168.11.6(출발)
0020  0b d3 86 1c 1c f3 00 a4 c2 56 52 54 50 53 02 02   .........VRTPS..   c0a80bd3=.211(도착) / 52545053="RTPS" / 0202=v2.2
0030  01 0f 01 0f b5 b4 7f 3f 1f fa 00 00 00 00 0e 01   .......?........   010f=vendor eProsima / 0e=INFO_DST
0040  0c 00 01 0f 52 7e e9 e8 20 5f 00 00 00 00 09 01   ....R~.. _......   09=INFO_TS
0050  08 00 e8 3b 3a 6a 7b d4 20 d3 15 05 2c 00 00 00   ...;:j{. ...,...   15=DATA 서브메시지 시작
0060  10 00 00 00 03 04 00 00 03 03 00 00 00 00 02 00   ................   0304=readerEID 0303=writerEID / seq=2
0070  00 00 00 01 00 00 02 00 00 00 0c 00 00 00 48 65   ..............He   0001=CDR_LE / 02000000=index2 / 0c000000=len12

```
→ 0x70~0x82 구간의 `02 00 00 00 | 0c 00 00 00 | 48 65 6c 6c 6f 20 44 44 53 20 32 00` 이 곧 **`index=2, "Hello DDS 2"`**. 오른쪽 ASCII에 그대로 "Hello DDS 2"가 보인다.

### 9-5. HEARTBEAT (프레임 19) — RELIABLE 신뢰성
```
INFO_DST → HEARTBEAT
  writerEntityId : (해당 Writer)
  firstAvailableSeqNumber : 1     ← "내가 가진 가장 오래된 시퀀스"
  lastSeqNumber           : 0/N   ← "가장 최신 시퀀스"
  count                   : 1     ← 하트비트 일련번호
```
→ Writer가 "1~N번까지 있다, 받았는지 알려줘"라고 통지. Reader는 이걸 보고 빠진 게 있으면 ACKNACK으로 요청.

**실제 원본 바이트 (프레임 19):**
```
0040  0c 00 01 0f 52 7e e9 e8 20 5f 00 00 00 00 07 01   ....R~.. _......   07=HEARTBEAT 서브메시지
0050  1c 00 00 00 04 c7 00 00 04 c2 00 00 00 00 01 00   ................   04c7=rdEID 04c2=wrEID(SEDP Sub)
0060  00 00 00 00 00 00 00 00 00 00 01 00 00 00 80 01   ................   firstSN=1 lastSN=0 count=1
```
→ `04c2`=BUILTIN_SUBSCRIPTIONS_WRITER, 그 뒤 `01..00`이 firstAvailableSeqNumber=1, lastSeqNumber=0, count=1.

### 9-6. ACKNACK (프레임 32) — 수신 확인/재전송 요청
```
INFO_DST → ACKNACK   Flags: Final
  readerSNState:
     bitmapBase : 1     ← "1번부터 기대 중"
     numBits    : 0     ← 비어있음 = 빠진 것 없음(전부 받음)
  Count : 1
```
→ `bitmapBase`까지는 다 받았다는 뜻. 빠진 시퀀스가 있으면 비트맵에 표시해 재전송 요청. **numBits=0 = 손실 없음.**

**실제 원본 바이트 (프레임 32):**
```
0040  0c 00 01 0f 52 7e e9 e8 20 5f 00 00 00 00 06 03   ....R~.. _......   06=ACKNACK / 03=Final+Endian 플래그
0050  18 00 00 03 01 c4 00 03 01 c3 00 00 00 00 01 00   ................   0301c4=rdEID 0301c3=wrEID
0060  00 00 00 00 00 00 01 00 00 00 80 01 38 00 01 00   ............8...   bitmapBase=1 numBits=0 count=1
```
→ `01 00 00 00 00 00 00 00`(bitmapBase=1) + `00 00 00 00`(numBits=0) = "1번부터 기대, 빠진 것 없음".

### 9-7. GAP (프레임 96) — 시퀀스 건너뜀 통지
```
GAP
  gapStart : 2          ← "2번은"
  gapList bitmapBase: 3 ← "3번부터로 건너뛴다(2번은 없음)"
```
→ KEEP_LAST로 잘려나갔거나 보낼 게 없는 시퀀스를 Reader에 알려, Reader가 그 번호를 무한정 기다리지 않게 함(정합성 유지).

**실제 원본 바이트 (프레임 96):**
```
0040  0c 00 01 0f b5 b4 7f 3f 1f fa 00 00 00 00 08 01   .......?........   08=GAP 서브메시지
0050  1c 00 00 00 04 c7 00 00 04 c2 00 00 00 00 02 00   ................   04c7/04c2=엔드포인트 / gapStart=2
0060  00 00 00 00 00 00 03 00 00 00 00 00 00 00 07 01   ................   gapList bitmapBase=3 / 07=뒤이어 HEARTBEAT
```
→ `gapStart=2`, `bitmapBase=3` = "2번은 건너뜀(없음), 3번부터 유효". 뒤에 HEARTBEAT가 붙어 한 패킷에 같이 옴.

> **SPDP `DATA(p)`(프레임 1)** 는 606바이트로 커서 전체 hex는 생략. 핵심은 본문에 위에서 본 PID 파라미터들(로케이터 IP/포트, QoS, 빌트인 엔드포인트 셋)이 TLV 형식으로 쭉 나열된 구조다. 전체는 `tshark -r <pcap> -Y "frame.number==1" -x` 로 확인.

### 9-8. 빌트인 EntityId 정리 (패킷에서 본 값)
| EntityId | 의미 |
|----------|------|
| `0x000100c2 / c7` | SPDP 참여자 Writer/Reader |
| `0x000003c2 / c7` | SEDP Publications(Writer 광고) |
| `0x000004c2 / c7` | SEDP Subscriptions(Reader 광고) |
| `0x000003xx` 등 (낮은 c2/c7 아님) | **애플리케이션(사용자) 엔드포인트** |

→ EntityId만 봐도 그 패킷이 **Discovery용(빌트인)인지 실제 데이터(사용자)인지** 구분된다.
